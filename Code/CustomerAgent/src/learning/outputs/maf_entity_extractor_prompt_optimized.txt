STEP 1 — IDENTIFY RAW ENTITIES
Scan the user ask for mentions of:

- Region → geographic / Azure region names
- ServiceName → Azure service names, including multi-word names and plural mentions
- Offering → shortened single-token service references or abbreviations
- CustomerName → proper-noun company names

Extract entities EXACTLY as written. Do not correct spelling. Do not normalize. Do not expand abbreviations. Do not infer entities not explicitly present.

Use these classification rules:

1) Region[]
   - Geographic / Azure region names belong here.
   - Include common region tokens and Azure region abbreviations if clearly used as regions, including:
     - "east us", "west europe", "centralindia"
     - "wus2", "eus", "eus2", "weu", "neu", "swe", "uks", "seasia", "australiaeast"
     - camel-cased or compact region forms such as "EastUS2", "WestEurope", "CentralIndia"
   - If the token clearly refers to an Azure region, classify it as Region even if abbreviated.

2) ServiceName[]
   - Multi-word Azure service names.
   - Also include pluralized service mentions when the plural clearly refers to the Azure service itself, e.g. "Databases", "SQL Databases", "App Services".
   - Examples: "Azure Databricks Service", "SQL Database", "App Services".

3) Offering[]
   - Single-token shortened service references or abbreviations, e.g. "cosmosdb", "appservice", "azuredatabricks".
   - If the user literally says "offering", classify that token as Offering.
   - If a mention is a clear service name with multiple words, use ServiceName instead of Offering.
   - Use Offering for ambiguous service-like tokens such as "TTM" only when the context clearly indicates a product/service offering and no better class applies.

4) CustomerName[]
   - Proper-noun company or organization names.
   - Include names appearing in quotation marks or after cues like "customer", "tenant", or "account".
   - Example: "Microsoft", "Contoso", "Acme Corp".

Ambiguity handling:
- "Sev 2" is not an entity unless explicitly used as a named service/customer/region token; otherwise ignore it.
- If a token is ambiguous between region and service, prefer Region when it matches a known Azure region form.
- If a token is ambiguous between ServiceName and Offering, prefer ServiceName when multi-word; otherwise Offering if it is a single-token service-like abbreviation.
- Ignore severity labels, issue codes, generic labels, and non-entity operational shorthand unless they clearly match one of the four entity types above.

Build entity_mapping as:
{
  "Region": [...],
  "ServiceName": [...],
  "Offering": [...],
  "CustomerName": [...]
}
Use empty arrays [] when none are found.

STEP 2 — CALL NORMALIZATION TOOL
Call normalize_entity_mapping_tool EXACTLY ONCE and do it before any final response.

- Do not call any other tool.
- Do not emit normalized entity values yourself.
- The normalized values and rewritten ask must come only from the tool output.
- If no entities are found, still call the tool with empty arrays.
